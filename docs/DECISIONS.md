# Decision log (build-time)

Honest record of scope and technology decisions made during the Phase 1 build, and why.

1. **Drizzle ORM instead of Prisma.** The build environment's network policy blocks `binaries.prisma.sh`, which Prisma needs for its native engines. Drizzle is pure JavaScript (no binary downloads), runs identically on Neon/Vercel, and is arguably a better serverless fit (smaller cold starts). No feature was lost; the schema is in `db/schema.ts`.
2. **Hand-rolled session auth (bcrypt + jose JWT cookie) instead of Auth.js.** Fewer moving parts for credentials-only Phase 1; passwords are bcrypt-hashed, sessions are signed HTTP-only `SameSite=Lax` cookies, and every server action re-verifies project access (`requireProjectAccess`). OAuth providers are a Phase 2 config task, not a rebuild.
3. **Server actions + server components instead of a REST API.** All reads happen in server components, all writes in `app/actions.ts` with auth checks. Cuts an entire API layer; if a public API is wanted later, `lib/` functions are already separated from transport.
4. **Charts: recharts + one custom SVG gauge.** Control chart needs σ zone bands and per-point flag styling — done with recharts primitives; the OEE gauge is simpler as hand-written SVG.
5. **System font stack.** Build environment cannot fetch Google Fonts at build time (`next/font/google` is a build-time download). Distinctiveness comes from the palette, the σ zone-band motif, and monospaced tabular numerals.
6. **Excel WE rules limited to 1 and 4.** WE2/WE3 ("2 of 3 beyond 2σ", "4 of 5 beyond 1σ") are genuinely awkward as maintainable spreadsheet formulas; shipping them half-right is worse than saying plainly (on the Guide sheet and next to the charts) that the app checks all four and Excel checks two.
7. **Pareto category list in Excel is manual.** Auto-extracting unique codes needs `UNIQUE` (breaks old Excel/LibreOffice and the verification pipeline). The app aggregates automatically; the sheet says so.
8. **PWA offline scope = read-only.** Claiming offline data entry without a sync queue would be a fake feature. The service worker caches visited pages network-first; offline entry is a labeled roadmap item.
9. **Auto-CAPA is created as a DRAFT.** The rules engine never silently commits a quality record; a human confirms. Same philosophy as playbook decision gates.
10. **"Dimac" in the source brief interpreted as DMAIC**, with the Define phase included; the brief's four phases (measure/analyze/improve/control) are a DMAIC subset.
11. **Renamed OpsForge → Procyra.** "OpsForge" was already in active use by multiple existing companies/products, including at least one direct competitor in the same manufacturing-operations space — a real collision, not a coincidental keyword hit. "Procyra" cleared a GitHub repo/org search and a general web search with no exact-name conflicts found. Renamed throughout: package name, README, all docs, UI copy, PWA manifest (name/short_name), session cookie name, and the three companion workbook titles/headers (regenerated from the updated `spreadsheets/src` generator scripts, not hand-edited — re-verified at 0 formula errors across all 940 formulas post-rename). `package-lock.json` was regenerated rather than hand-patched to avoid corrupting integrity hashes.
12. **Documented the `tsx` env-loading gotcha.** `npx tsx scripts/e2e.ts`, run exactly as originally documented, fails with a confusing Postgres "no user specified" error because `tsx` does not auto-load `.env` the way `next dev`/`next build` do. This is a documentation bug, not a logic bug — the automation loop itself was already correct. README now documents the working alternative (`node --env-file=.env node_modules/.bin/tsx scripts/e2e.ts`), verified working with env vars unset beforehand.
